## AI Service Architecture

The AI service layer was implemented using a modular structure that separates concerns across multiple components.

### External API Integration

The system integrates with the Groq API using the `llama-3.3-70b-versatile` model. API communication is handled through the `groq` Python library, following the standard chat completion interface.

The API key is loaded securely from a `.env` file using `python-dotenv`, ensuring that sensitive credentials are never hardcoded into the source code.

### Prompt Engineering

The `_build_prompt()` function constructs a structured prompt that includes:

- Product name
- Current stock level
- Average daily sales (last 7 days)
- Estimated days remaining before stockout
- Urgency status (CRITICAL / LOW / NORMAL / OVERSTOCK / NO SALES)

The AI is instructed to respond as an experienced coffee shop operations consultant, providing concise and actionable recommendations in plain English.

### Error Handling

All external API calls are wrapped in `try-except` blocks. In case of API failure, quota exhaustion, or network errors, the system returns a user-friendly warning message instead of crashing the backend. This ensures system stability under all conditions.

## Rate Limiting and Caching

### Rate Limiter

The `ai_rate_limiter.py` module implements a sliding window rate limiter that enforces:

- A maximum of 14 requests per minute
- A maximum of 1400 requests per day

These limits are set slightly below the Groq free tier limits to provide a safety buffer.

### In-Memory Cache

The caching system stores AI responses for 5 minutes using a TTL (Time-To-Live) strategy. When the same product with the same stock level and average sales is requested again within the cache window, the system returns the cached response without making a new API call.

This significantly reduces API usage and improves response times for repeated queries.

## Database Bridge

The `ai_db_bridge.py` module connects the AI service layer with the database models implemented by the Database Engineer.

Key functions include:

- `get_avg_daily_sales()` – calculates the average daily sales for a product over the last 7 days using SQLModel queries.
- `get_product_with_ai_inputs()` – retrieves a single product with all data required for AI analysis.
- `get_all_products_with_ai_inputs()` – retrieves all active products ready for bulk AI analysis.

The module uses a `try/except ImportError` pattern to fall back to stub models when the database models are not yet available, enabling independent development and testing.

## API Endpoints

The following FastAPI endpoints were implemented in `ai_router.py`:

| Method | Endpoint | Description |
|--------|----------|-------------|
| GET | `/api/ai/analyze/{product_id}` | Analyze a single product by database ID |
| POST | `/api/ai/analyze` | Analyze a product with manually provided data |
| GET | `/api/ai/analyze/bulk/all` | Analyze all active products at once |
| GET | `/api/ai/status` | View rate limiter and cache statistics |
| DELETE | `/api/ai/cache/{product_name}` | Invalidate cached advice for a product |

## Reflection

Implementing the AI integration layer required careful consideration of reliability, security, and efficiency. External API services can be unavailable or rate-limited at any time, so building a resilient service layer that degrades gracefully was a key design priority.

The combination of in-memory caching and rate limiting ensures that the system remains stable and cost-effective even under high usage scenarios.

By structuring the AI layer as an independent module with clear interfaces, the system supports parallel development across team members without blocking dependencies.
