## Database Architecture

The database layer was implemented using **SQLModel**, an ORM that combines the capabilities of SQLAlchemy and Pydantic to provide type-safe database models and automatic data validation.

### Core Entities

- **Product** – central inventory entity

- **Sale** – historical sales data

- **Waste** – inventory loss tracking

### Extended Entities

- **Supplier** – represents product suppliers

- **PurchaseOrder** – represents restocking operations

- **PurchaseOrderItem** – links products to purchase orders

These entities form a relational structure that supports realistic inventory modeling.

## Data Initialization & Simulation

- `init_db.py` initializes the database and inserts structured base data

- `seed_sales.py` generates ~30 days of sales with dynamic demand patterns

- `seed_waste.py` simulates inventory loss

These scripts provide realistic historical data for analysis.

## System Architecture and Integration

### Inventory Simulation and Stock Management

To improve the realism of the system, I implemented a stock simulation mechanism that enables dynamic inventory updates instead of static stock values.

This functionality introduces a complete inventory lifecycle by modeling daily operations such as sales, waste, and restocking.

The following API endpoints were developed to support this behavior:

- **POST /api/stock/reset-stocks**  
  Initializes inventory with predefined stock values  

- **POST /api/stock/simulate-day**  
  Simulates daily sales and waste, updating stock levels accordingly  

- **POST /api/stock/restock/{product_id}**  
  Allows manual restocking of products  

These operations are tightly integrated with the database layer using SQLModel and ensure that inventory data is continuously updated.

This improvement allows the AI module to generate recommendations based on realistic and dynamic data rather than static values.
