## Database Architecture

The database layer was implemented using **SQLModel**, an ORM that combines the capabilities of SQLAlchemy and Pydantic to provide type-safe database models and automatic data validation.

The system uses **three relational tables**:

**Product**

Represents inventory items available in the coffee shop.

Key attributes:

- `id` – primary key

- `name` – product name

- `current_stock` – current inventory level

- `unit` – measurement unit (kg, liter, piece)

This table acts as the central entity of the inventory system.

**Sale**

Stores historical sales records used for demand analysis.

Attributes:

- `id`

- `product_id` (foreign key → Product)

- `quantity`

- `date`

Relationship:

```bash
Product (1) → (N) Sale
```

This structure allows the system to calculate average sales velocity, which is later used in AI-based stock recommendations.

**Waste**

Tracks inventory loss due to expiration or operational waste.

Attributes:

- `id`

- `product_id` (foreign key)

- `quantity`

- `date`

- `reason`

Relationship:

```bash
Product (1) → (N) Waste
```

This data helps simulate real-world inefficiencies in inventory management.

### Additional Backend and Database Integration Tasks

During the later stages of the project, I further contributed to the database layer by implementing additional backend functionality required to support dynamic inventory behavior and system integration.

One of the key improvements was the implementation of a stock simulation mechanism. Previously, inventory values remained static, which limited the effectiveness of the AI-based recommendation system. To address this issue, I implemented backend endpoints that allow the system to dynamically update inventory levels based on simulated daily operations.

The following endpoints were implemented and integrated with the database layer:

• **POST /api/stock/reset-stocks**
Initializes the inventory with predefined stock values for all products.

• **POST /api/stock/simulate-day**
Simulates daily sales and waste activity and automatically decreases product stock levels.

• **POST /api/stock/restock/{product_id}**
Allows manual restocking of a specific product.

These endpoints were implemented using FastAPI routers and interact directly with the SQLModel-based database layer to update product inventory levels.

### Database Configuration Improvements

While integrating the database with the backend service, I also improved the database configuration to ensure consistent behavior across different environments.

The SQLite connection was modified to use absolute path resolution when locating the database file. This prevents the accidental creation of multiple database files in different working directories and ensures that the backend always connects to the correct database instance.

This modification improves reliability and prevents data inconsistencies during development and testing.

### Environment Configuration for AI Integration

To support the AI analysis module, environment variable configuration was implemented using a .env file. This allows sensitive credentials such as API keys to be securely stored outside of the source code.

The backend service reads the API key from environment variables when communicating with the AI analysis service. This follows common backend development practices by separating configuration from application code and preventing sensitive credentials from being included in the repository.
